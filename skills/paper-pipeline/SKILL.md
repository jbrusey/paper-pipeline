## Goal

Maintain a paper pipeline in which every derived artefact is generated from source data, configuration files, or scripts, and the final paper is built automatically from Markdown into LaTeX/PDF.

## Core principles

* Do not hard-code numerical results in prose.
* Generate all reported values from scripts.
* Build every paper component through `make`.
* Keep raw data, processed data, figures, tables, and generated LaTeX macros separate.
* The final manuscript should be rebuildable from a clean checkout.

## Recommended structure

```text
paper/
  Makefile
  paper.md
  references.bib
  config/
    hyperparameters.yaml
  data/
    raw/
    processed/
  scripts/
    generate_metrics.py
    generate_tables.py
    generate_figures.py
  build/
    values.tex
    tables/
    figures/
    paper.tex
    paper.pdf
```

## Generated numerical values

Numerical values used in prose, such as percentage improvements, error rates, sample counts, or confidence intervals, should be generated as LaTeX macros.

Example output in `build/values.tex`:

```latex
\newcommand{\AbstractImprovement}{\SI{12.4}{\percent}}
\newcommand{\BestRMSE}{\num{0.83}}
\newcommand{\NumParticipants}{\num{42}}
```

The manuscript should then use:

```latex
Our method improved performance by \AbstractImprovement{} compared with the baseline.
```

Values should normally come from source files such as:

* `config/hyperparameters.yaml`
* `data/processed/results.csv`
* experiment logs
* model evaluation summaries

## Tables

Tables should be generated by scripts, preferably as LaTeX tables.

Use Python tools such as:

* `pandas`
* `tabulate`
* `siunitx` formatting

Numerical values in tables should be formatted for LaTeX using `siunitx`, for example:

```latex
\num{0.843}
\SI{12.4}{\percent}
```

Avoid manually editing generated table files.

## Figures

Figures should be generated from scripts.

Preferred formats:

* SVG for line plots, scatter plots, and diagrams with fewer than around 100 data points.
* PNG for dense raster-like plots, heatmaps, or very large datasets.
* PDF may be used for LaTeX-native publication workflows where appropriate.

Use matplotlib unless there is a clear reason to use another plotting library.

Figure scripts should:

* read from data or config files;
* write to `build/figures/`;
* use deterministic settings where possible;
* avoid manual post-processing.

## Makefile requirements

The `Makefile` should define targets for each generated component.

Typical targets:

```make
all: build/paper.pdf

build/values.tex: scripts/generate_metrics.py data/processed/results.csv
	python scripts/generate_metrics.py

build/tables/results.tex: scripts/generate_tables.py data/processed/results.csv
	python scripts/generate_tables.py

build/figures/main_result.svg: scripts/generate_figures.py data/processed/results.csv
	python scripts/generate_figures.py

build/paper.pdf: paper.md build/values.tex build/tables/results.tex build/figures/main_result.svg references.bib
	pandoc paper.md \
	  --from markdown \
	  --to latex \
	  --output build/paper.pdf \
	  --bibliography references.bib \
	  --citeproc \
	  --include-in-header build/values.tex

clean:
	rm -rf build/*
```

## Markdown manuscript

The main manuscript should be written in Markdown.

It may include generated LaTeX components directly:

```markdown
# Results

Our approach achieved an improvement of \AbstractImprovement{}.

\input{build/tables/results.tex}

![Main result](build/figures/main_result.svg)
```

## Codex working rules

When using Codex:

1. First inspect the `Makefile`.
2. Do not edit generated files directly.
3. If a value, table, or figure is wrong, edit the source data, config, or generation script.
4. After changes, run the relevant `make` target.
5. Prefer small, testable changes.
6. Ensure the full paper builds with `make all`.
7. Keep generated artefacts out of version control unless the project explicitly requires them.
8. Preserve reproducibility over convenience.

## Validation checklist

Before considering the paper complete:

* `make clean && make all` succeeds.
* All numerical values in prose are generated.
* All tables are generated.
* All figures are generated.
* `siunitx` is used for numerical formatting in LaTeX.
* No generated file has been manually edited.
* The final PDF matches the intended manuscript.
* Dependencies are documented.
* Random seeds are fixed where relevant.

## Preferred behaviour

When asked to add or update a result:

* identify the source of the result;
* update the relevant script or data source;
* regenerate the LaTeX macro, table, or figure;
* rebuild the paper;
* report exactly which files changed.
